# 编码规范（严格）

## 元信息
- 版本：v1.1
- 生效日期：2026-02-28
- 最后更新：2026-03-01
- 责任人：reviewer-agent
- 变更说明：补充元信息并对齐规范基线。


## 目标与范围
- 目标：保证可维护性、安全性与一致性。
- 适用范围：所有生产代码、脚本与配置即代码。

## 代码组织
- 模块边界必须清晰，职责单一。
- 公共接口必须有文档或注释说明。
- 禁止跨层直接访问内部实现。

## 命名与风格
- 文件、目录、模块名必须语义明确。
- 公共 API 必须保持统一命名风格。
- 禁止拼音、含混缩写与临时命名。

## 错误处理
- 所有可预期错误必须显式处理。
- 错误信息不得泄露敏感信息。
- 关键错误必须有统一错误码与告警。

## 依赖与复用
- 禁止复制粘贴未知来源代码。
- 第三方依赖准入、许可、SBOM 与供应链控制以 `specs/standards/S08-可信合规.md` 为准。
- 复用公共能力优先于重复实现，必要重复必须记录原因。

## 日志与可观测性
- 指标、日志、追踪与告警基线以 `specs/standards/S09-运营韧性.md` 为准。
- 代码实现必须提供关键路径埋点并保持可关联请求标识。

## 代码评审
- 关键模块必须评审通过方可合并。
- 评审必须记录结论与问题清单。

## 审计与证据
- 必须保留执行证据（评审记录、报告、清单或日志）。
- 证据必须关联到对应任务、变更或发布记录，确保可追溯。

## 例外处理
- 任何例外必须形成 ADR 并注明补偿措施。
