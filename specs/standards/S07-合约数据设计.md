# 接口与数据设计规范（严格）

## 元信息
- 版本：v1.0
- 生效日期：2026-03-02
- 最后更新：2026-03-02
- 责任人：architect-agent
- 变更说明：合并 api-design 与 data-model，统一需求弱设计强场景下的契约与模型标准。

## 目标与范围
- 目标：确保接口契约与数据模型语义一致、可验证、可演进。
- 适用范围：HTTP/gRPC/消息接口契约、JSON Schema、关系/文档/事件模型设计。

## 设计边界与引用
- 需求与设计边界以 `specs/standards/S03-文档规范.md` 为准。
- 发布兼容与回滚要求以 `specs/standards/S05-交付控制.md` 为准。
- 安全、隐私与依赖约束以 `specs/standards/S08-可信合规.md` 为准。

## 需求映射要求
- 每个公开接口必须回链至少一个需求条目（`RQ-*` 或 `API-CAP-*`）。
- 每个核心实体必须映射到需求中的业务对象或规则条目（如 `DATA-RULE-*`）。
- 接口语义或数据约束不得凭空新增；新增项必须先更新需求或 ADR。
- 契约与模型变更必须同步更新追踪关系与发布记录。

## 接口契约设计要求
- 必须定义输入、输出、错误码、幂等性与超时语义。
- 必须定义鉴权方式、权限边界与审计字段。
- 必须给出失败路径与重试/补偿策略。
- 设计文档应记录接口分组、调用方、限流与错误语义。

## 数据模型设计要求
- 必须定义实体、属性、主键/唯一性、引用关系与生命周期。
- 必须定义一致性要求（强一致/最终一致）与事务边界。
- 必须定义审计字段、软删/硬删策略与保留周期。
- 关键模型必须关联测试数据策略与合规要求。

## 演进与兼容策略
- 破坏性契约或模型变更必须提升主版本并提供迁移策略。
- 非破坏性新增必须保持旧消费者可用，历史数据与旧接口可读取。
- 废弃接口或模型必须定义观察期、下线计划与数据清理计划。
- 破坏性变更必须提供迁移脚本与回滚路径。

## 文档与产物
- OpenAPI 契约位于 `specs/2-designs/api/*.openapi.yaml`。
- JSON Schema 位于 `specs/2-designs/schema/*.schema.json`。
- 每个关键接口与模型必须可追踪到测试用例与发布记录。

## 检查与验收
- 每个接口/模型必须具备需求映射、完整定义与兼容策略三项证据。
- 未完成映射、未定义兼容策略或未说明迁移回滚的变更不得进入发布。
- 重大接口或模型变更必须通过评审，必要时形成 ADR。

## 工具支持
- `python3 specs/meta/tools/sddtool.py validate-design`：检查设计文档关键章节。
- `python3 specs/meta/tools/sddtool.py check-completeness`：检查 RQ-DS-TK-TEST 链路。
- `python3 specs/meta/tools/sddtool.py check-naming`：检查 API/Schema 产物命名与索引登记。

## 审计与证据
- 必须保留执行证据（评审记录、报告、清单或日志）。
- 证据必须关联到对应任务、变更或发布记录，确保可追溯。

## 例外处理
- 接口或数据模型例外必须形成 ADR 与批准记录。
