# 治理与流程（严格）

## 1. 治理目标与原则
- 保障 specs 为唯一事实源（SSF）。
- 将需求、设计、实现、测试、发布形成闭环。
- 保持证据可追溯、变更可审计、风险可控。
- 严格控制范围蔓延与不可验证需求。
- `specs/govs/`（Policy）定义责任、审批层级与升级路径。
- `specs/standards/`（Standard）定义执行方法、技术约束与操作清单。

## 2. 决策与审批层级
- 产品/业务 Owner（人类角色）：定义目标、范围与验收标准，批准关键决策。
- Spec 主编（人类角色）：维护 specs 体系一致性与完整性。
- L1（局部实现）：developer-agent 与 Spec 主编确认。
- L2（跨模块设计）：architect-agent 与 Spec 主编确认。
- L3（重大架构/安全/合规）：Owner 批准并生成 ADR。
- L4（范围/预算/交付重大变化）：Owner 与 reviewer-agent 联签。
- 角色细则与 RACI 见 `specs/govs/G04-角色职责.md`；Agent 与 Skill 映射以 `specs/meta/agents/agents.md` 为准。

## 3. 变更控制
### 3.1 变更分级
- 标准变更：低风险、可快速回滚。
- 重大变更：涉及架构、安全、合规或关键链路。

### 3.2 变更流程
1. 提交变更申请并说明动机。
2. 评估影响范围、风险和回滚。
3. 按分级进行审批。
4. 执行变更并回填证据。
5. 完成回归与复核。

### 3.3 决策门槛
- 每项决策必须列出 2-4 个可行选项。
- 必须记录目标、约束、证据、成本与风险。
- 重大决策必须有 ADR。

### 3.4 强制要求
- 变更必须关联需求/设计/任务。
- 重大变更必须形成 ADR。
- 未通过回归验证不得放行发布。
- 执行侧要求与操作清单以 `specs/standards/S05-交付控制.md` 为准。

### 3.5 文档操作规范（强制）
#### 3.5.1 必须通过工具执行CRUD
所有 specs/ 目录下的文档操作（创建、读取、更新、删除）**必须通过 CLI 工具执行**，禁止直接手动操作文件。

| 操作 | 禁止方式 | 必须方式 |
|------|----------|----------|
| **创建文档** | 直接复制/新建文件 | `sddtool.py create-<type>` |
| **读取文档** | 直接打开文件查看 | `sddtool.py read-document <ref-id>` |
| **更新文档** | 直接编辑修改 | `sddtool.py rename-document`（更换编号）或走变更流程 |
| **删除文档** | 直接删除文件 | 暂不支持删除，需通过 Spec 主编审批后手动执行 |
| **检索关联** | 手动搜索目录 | `sddtool.py trace-dependencies <ref-id>` |
| **定位文档** | 按文件名查找 | `sddtool.py locate-document <ref-id>` |

#### 3.5.2 工具使用规范
- **文档编号唯一性**：脚本自动校验引用编号（如 RQ-10102）在 `specs/` 目录下的唯一性。
- **SLUG解耦**：通过文档编号定位文档，无需携带SLUG信息。SLUG变更不影响脚本操作。
- **重复编号处理**：如果同一编号找到多个文档，必须先用 `rename-document` 更换其中一个的编号。
- **代码追溯**：通过 `trace-dependencies` 命令可检索关联的需求、设计、任务及代码实现位置（含行号、模块/函数名）。

#### 3.5.3 违规后果
- 直接手动操作文档视为违反治理规范。
- 手动操作导致索引不一致、追溯断裂的，必须回滚并重新通过工具执行。
- 审计发现的手动操作将记录在合规报告中。

### 3.6 项目文件夹管理规范（强制）
#### 3.6.1 文件夹整洁原则
项目文件夹为高度整齐的受控环境，禁止未经确认的文件创建。

#### 3.6.2 临时文件管理
- **唯一临时目录**：`tmp/` 为唯一允许生成临时报告和临时文件的目录。
- **临时文件用途**：分析报告、中间结果、调试输出、临时数据。
- **清理责任**：临时文件不纳入版本控制，使用者负责定期清理。

#### 3.6.3 文件创建权限控制
| 目录 | 允许操作 | 禁止操作 | 例外处理 |
|------|----------|----------|----------|
| `specs/` | 通过工具创建（见 3.5.1） | 直接创建文件 | 必须经 Spec 主编确认 |
| `tmp/` | 自由创建临时文件 | 提交到版本控制 | 无 |
| 其他目录 | 无 | 未经确认创建文件 | 必须经 Owner 或 Spec 主编确认 |

#### 3.6.4 Agent 行为约束
- Agent 生成任何报告或文档前，必须询问用户确认存放位置。
- 默认存放位置为 `tmp/`，其他位置需用户明确授权。
- 禁止在项目根目录或其他受控目录随意创建文件。

#### 3.6.5 违规后果
- 未经确认在受控目录创建文件视为违反治理规范。
- 违规文件必须立即移除或移动到 `tmp/`。
- 重复违规将记录在合规报告中并升级处理。

## 4. 评审与审计
### 4.1 评审制度
- 评审范围：需求、架构、代码、发布必须评审；高风险变更必须附加专项评审。
- 评审组织：评审必须指定责任人与参与角色；关键评审必须包含 Owner 与评审 Agent。
- 评审要求：结论必须记录并固化到 spec 或 ADR；未解决问题不得进入下一阶段。
- 评审输出：通过 / 有条件通过 / 不通过；问题清单需含责任人、截止时间、验证方法。
- 评审例外必须形成 ADR 与批准记录。

### 4.2 审计制度
- 审计范围：spec 完整性、证据可追溯性、实现一致性、变更与发布合规性。
- 审计频率：里程碑审计、发布前审计、重大事故后专项审计。
- 审计方法：文档一致性检查、追踪矩阵验证、变更记录与回滚策略核对。
- 审计结论必须记录并形成整改任务；高风险问题必须阻断发布。
- 审计例外必须形成 ADR 与批准记录。

## 5. 安全治理
- 治理目标：在需求、设计、实现、发布全流程落实安全控制。
- 治理要求：关键功能必须进行威胁建模或等价评估；依赖供应链必须进行漏洞与来源审查。
- 安全门禁：
  - 设计阶段：安全约束入设计与 ADR。
  - 实现阶段：静态检查与依赖扫描通过。
  - 发布阶段：高危漏洞清零或有批准例外。
- 事件响应：重大安全事件必须触发专项复盘并回填规范与任务。
- 安全例外须明确期限、责任人与补偿措施，到期必须复审。

## 6. 政策生命周期管理
- 制定与更新：治理与标准文档必须版本化与可追踪；重大更新必须评审并记录原因。
- 评审节奏：至少每 12 个月复审一次；重大事故或合规变化必须触发复审。
- 发布与生效：政策发布必须记录生效日期与适用范围；旧版本必须归档且可追溯。
- 例外与豁免：必须记录期限、范围、补偿控制、责任人与回收日期。
- 废止与替换：必须说明替代方案与迁移计划。

## 7. 里程碑与质量闸门
- 重要讨论必须形成结论并写入 spec 或 ADR。
- 讨论记录只作为临时材料，最终结论必须固化。
- 高风险变更必须有回滚方案与验证步骤。
- 所有安全与合规风险必须有 ADR 记录。
- 关键风险必须有监控指标与告警策略。
- Spec 闸门：`1-reqs/`、`adrs/`、`2-designs/`、`3-tasks/` 必须齐备。
- 证据闸门：外部事实必须记录在 `specs/meta/index/sources.md`。
- 实现闸门：实现与验收标准一致，测试通过且无高危缺陷。
- 发布闸门：发布说明、回滚策略、风险评估齐备。
- M0 需求冻结：完成需求与范围，澄清问题清零。
- M1 设计冻结：架构与关键 ADR 完成。
- M2 任务冻结：任务计划与验收标准完成。
- M3 发布候选：测试与安全审计通过。
