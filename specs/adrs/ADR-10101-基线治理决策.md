# ADR：基线治理与校验收敛

## 元信息
- 标题：校验规则从“仅结构”升级为“结构 + 实质内容 + 实体存在性”
- 状态：通过
- 日期：2026-03-02
- 负责人：reviewer-agent

## 背景与问题
- 背景：模板仓库已具备完整目录与命令，但核心文档存在占位内容时仍可能通过部分检查。
- 问题：缺少对“追踪 ID 是否真实存在工件”的自动阻断，导致链路可被文本占位伪造。
- 约束：保持 Python 标准库实现，不引入外部依赖。

## 决策驱动因素
- 目标：提高 SDD 文档可执行性与可审计性。
- 优先级：P0。
- 关键风险：规则增强会提高存量文档整改成本。

## 可选方案
- 方案 A：保持现状，仅人工评审识别占位内容。
- 方案 B：只加强 `validate-requirement/design`，不改完整性检查。
- 方案 C：同时加强 `validate-*`、`check-status`、`check-completeness`（采纳）。

## 决策
- 选择：方案 C。
- 理由：同时覆盖“内容占位”和“链路失效”两类核心质量风险，且不改变现有 CLI 命令面向用户的使用方式。
- 放弃原因：方案 A/B 无法阻断链路伪通过。

## 影响与后果
- 正向影响：文档质量门槛提升，追踪矩阵可信度更高。
- 负面影响：初次迁移需要补齐历史文档与 ID 映射。
- 影响范围：`specs/` 主工件、`sdd` 校验器、测试用例。

## 风险与缓解
- 风险：团队短期内因规则升级频繁阻断。
- 缓解措施：提供基线示例文档与清单化整改顺序。

## 迁移与回滚
- 迁移策略：先补齐主工件内容，再启用增强校验，最后在 CI 固化。
- 回滚策略：若发现误判，回退到前一版本校验实现并登记新 ADR。
